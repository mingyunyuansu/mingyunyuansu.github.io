总的来说，这一俩星期学的不算轻松，也很慢，因为很多是全新概念，每完成一两个步骤前都要读一两篇论文。不过很多内容其实本科时学校已经交给了我，只不过我不知道怎么使用他们而已，现在机会终于来了。这种感觉很奇妙，就好像学了好多年数学，现在开始学物理了那样，当汇编和堆栈那些看起来毫无用处的知识开始展现威力的时候感觉真的挺棒的。我一直担心对这个专业自己会不会是叶公好龙，等到真的去了华威后悔莫及。还好没有。尽管我依然觉得自己不是那种天生与计算机亲近的人，但至少学习信安技术让我觉得有意思，通过巧妙地计算和利用，让看起来随意的漏洞变得致命起来，简直魔法一样，很酷。信安学得好，他一定对整个计算机相关的所有知识都有认识，这点与我先前估摸的一样，很难，我就但行好事就行了吧。

因为是魔法系列episode 1，废话多了点，赶紧正题。本文叙述最经典的缓冲区溢出攻击，如你所想，这种攻击有不少复杂的变种，但是经典溢出攻击的思路是非常简单清晰的。

首先明确，尽管因OS内核版本，CPU型号，各种特殊情况不同，一个程序运行时的结构会有不同，但是总的来说基本是下面这样子：

Local variables | EBP | Return Address | Parameters (栈尾在此)

举个例子：

  • 当运行func(1, 2)函数前，程序会按照backwards的顺序，把int 2, int 1 push进栈中，作为parameters，即实参；
  • 随后立即运行汇编call func，把return address（即IP）push入栈，作为func结束运行后的返回点，重新回到main，这也叫flow control；
  • 之后便进入func内部，程序会立即运行以下汇编代码push ebp, mov esp ebp。就是把当前还“身处”main函数中的EBP（x86体系中的栈帧指针FP）入栈以便后续恢复，把“身处”main的ESP（x86的栈顶指针SP）赋值给EBP，相当于新建一个空栈，此时EBP=ESP，至此函数方可运行。这一过程被称为称为prelude；
  • 接下来显然要为local variables分配空间，即此处的str[10]，所以大概会执行sub 0xa, esp这样的代码，把栈顶指针移十个字节，栈空间增加十个字节。注意，Intel CPU还有现在很多主流操作系统的栈方向都是与内存相反的，也就是说栈是从高地址往低地址延伸，所以想扩大栈，是对栈顶指针做减法。其次，计算机分配空间只能是字大小的整数倍，32位机是4 bytes = 1 word，所以实际的代码将是sub 0xc, esp，三个words。（再再再注意，3个字是理论上，实际上我运行时发现机器很有可能再多分配几个，比如sub 0x10, esp什么的。）
  • 在运行return之前，会执行汇编代码leave，这语句从前是宏，现在被收为一条指令了，作用是mov ebp, esp, pop ebp。即把EBP恢复到调用func前的状态。但是ESP为什么要从EBP处得到值我就不太懂了，被调用的中EBP会怎么动根本说不清吧，这个值有什么意义吗？另外恢复EBP对main函数有什么用我现在也没看出来，在return to libc攻击中甚至会把EBP直接抹去，却装作无事发生。。。
  • 最后执行ret指令，众所周知，pop出return address，jmp过去，继续main函数

缓冲区溢出攻击，指的就是当程序读入过量数据，超过了Local variable区域的界限，覆盖了后面的栈内容时（外网称smashing the stack)，通过精心计算，让已提前存放于内存中的恶意程序的地址刚好覆盖掉return address，控制流就傻乎乎的跳转过去继续执行恶意代码了（当然恶意代码也是这个时候覆盖进来的，某种程度上是在栈里面）。

以上是理论基础，知道了程序的结构就可以进行定点的计算了。比如上面代码中，str只有大约十字节大小，如果程序用fread, gets, strcpy, scanf之类不检查边界情况的函数读100字节数据，就很有可能发生溢出，如果是恶意程序，攻击就启动了。一般来讲这个恶意程序会是一个简单的调用shell程序，如果这个时候运行者是root，那么我们就直接得到了一个有root权限的shell（但是操作系统往往会在这种调出的shell里自动drop掉root权限，这是一种防御机制，以后再说）

但是这么简单的攻击往往不会奏效，因为现在操作系统针对溢出攻击有非常多的防御机制，其中之一就是禁止运行栈。最近的Linux的编译器已经默认禁止运行栈了，我们的代码是存在栈里面的某个位置的，操作系统不让运行的话跳转过去也无济于事。于是我们有了return to libc攻击。

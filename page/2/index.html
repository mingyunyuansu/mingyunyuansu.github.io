  • 350.Intersection of Two Arrays II
    这题本身技术含量不高，不过我发现一个蛋疼的地方就是C语言标准库的qsort似乎有bug（至少LeetCode用的编译器是这样）。比如一个数组[-2147483648, 1, 2, 3]，如果用qsort按升序排序，那么结果中-2147483648会被当作一个比较大的正数，至少这里被当做比3还大。显然-2147483648是4字节带符号int的最小值，而且经测试leetcode的C编译环境就是一个int占4字节。所以猜测应该是把1000···000（补码最小数）当成了无符号数，which is 一个很大的正数，然后排序出错。当然啦我觉得更可能是我自己哪里搞错了，毕竟这年头这种级别的标准库会犯这种低级错误实在是难以想象。。。最后没办法换用C++然后用sort才正常实现。
  • 234.Palindrome Linked List
    就是判断一个链表是否是回文，不过题目要求在时间复杂度O(n),空间复杂度O(1)条件下完成。从别人的答案中学到了一个很有意思的办法，就是通过速率不同的双指针遍历线性数据结构。

我发现这种用快慢指针在遍历链表时用途很广，比如判断链表是否有环。
不过这题也有一个大神提出，任何修改了输入链表（也就是reversing）的办法其实都是空间复杂度O(N)，而不是题目要求的O(1)。space complexity O(1)在理论上是不可实现的，如下：

It is a common misunderstanding that the space complexity of a program is just how much the size of additional memory space being used besides input. An important prerequisite is neglected the above definition: the input has to be read-only. By definition, changing the input and change it back is not allowed (or the input size should be counted when doing so). Another way of determining the space complexity of a program is to simply look at how much space it has written to. Reversing a singly linked list requires writing to O(n) memory space, thus the space complexities for all “reverse-the-list”-based approaches are O(n), not O(1).
Solving this problem in O(1) space is theoretically impossible due to two simple facts: (1) a program using O(1) space is computationally equivalent to a finite automata, or a regular expression checker; (2) the pumping lemma states that the set of palindrome strings does not form a regular set.

gdb使用笔记

我一直都在寻找一个适合自己的调试工具：

  • VC6, code block太丑，也不简便
  • Visual Studio虽然功能爆炸，但是固态硬盘启动都要数秒钟，创建项目麻烦，功能眼花缭乱
  • Clion用过一段时间，好用到让人害怕，但是student licence快过期了，而且同样太过臃肿，我又不写工程

    最后还是回到了editor编辑，gcc编译，printf大法人肉debug，实在不行再开VS调试。学了汇编之后，感觉汇编那个debug工具是真好用，再一次萌生寻找新工具的想法。今天终于找到了，那就是gdb，跟vim同级的上古神器，但是不像vim那么难学（虽然学会了vim确实很好用），gdb的基本功能用起来相当直观而且人性化。

要使用gdb，则在编译文件时要加 -g 参数，生成符号表

  1. l (list), 显示指定行号或指定函数附近的代码
  2. r (run), 开始运行程序，直到程序结束或碰到断点
  3. b (break), 设置断点，可行号，可函数名，如b 12`b main`
  4. c (continue), 从停顿处继续执行到结束或断点，注意程序开始运行时只能用r
  5. n (next), 执行一条语句，但是会把调用函数当作一个整体，也就是“不进去”
  6. s (step), 单步，执行一条单语句，遇到函数会进入函数内部。另外在gdb中直接回车默认执行上一条命令，因此单步调试也很方便
  7. u (until), 执行到指定行号或指定函数开头（也是必须开始运行了才能用）
  8. p (print), 打印一个变量值，如 print i, 如果是数组会打印整个数组，非常强大
  9. disp (display), 把一个表达式设置为display，如 disp i每当程序停下来时自动print该值
  10. cl (clear), 取消断点，和b的格式相同，若该处有多个断电则同时取消
  11. i (info), 可显示很多东西的信息，如i b，显示断点信息，i disp显示display信息，i lo显示所有局部变量
  12. bt (back trace), 显示当前运行时信息
  13. d (delete), en(enable), 删除单个display用
    上面只是gdb常用功能中的一部分，gdb还有很多变态的功能，我一时半会儿用不到。另外在gdb中还可以自由调用函数，比如你在二叉树的题目，那么写一个打印二叉树的函数，就可以在调试的过程中打印二叉树(call my_func()), 很方便。

8086汇编冷知识

呃，其实就是学习汇编的时候总结的一些比较细的，但是又可能有用的知识整理，还有一些初学者的技巧，并不真的是生僻知识。

现在的操作系统多为64位，那么是运行不了debug.exe masm.exe link.exe的，而且从Windows 2000(还是多少？不太确定，反正不关键)后，微软也把dos给去掉了。所以现在写汇编非常麻烦，我是windows10 64位系统，需要打开一个dosbox软件模拟dos环境，然后挂载程序(比如源文件，debug，masm，link程序所在的盘)，然后在虚拟环境下运行调试。而且masm6.15的可能有一些奇奇怪怪的问题，换成masm5.0就应该会稳定许多。当然还有其他的工具，比如masm32，甚至是IDE都有。不过既然是学汇编，在学语法的同时还是运用原始的工具比较好，不能太依赖开发环境。

  • 在 debug 模式下与源程序中对代码的识别是略微不同的。比如：

    1. debug 中，mov ax, [0]就是把ds:0的内容复制到ax中，但是如果这么写到源代码中，编译器会把[0]当做一个立即数0. 所以在源代码中，应该把内存的地址放到bx寄存器中（而且只能是bx），引用，如mov ax, [bx]。或者显式地指出短地址，比如写成mov ax, ds:[0]。注意，源代码中的mov ax, [bx]实际上是编译器默认数据段地址使用ds
    2. debug 中，输入的数字全部默认是16位，但是源代码中，要输入十六位数字必须末尾加h，否则编译器默认是十进制数

  • 在 debug 中快速越过循环：

    1. 可以在下一条命令是loop 时，按p，这时程序将会执行到 cx = 0 为止，即loop的下一条指令
    2. 或者使用 g 命令，直接跳到指定的偏移地址，代码段由 cs给出，比如 g 0012，将直接执行到 cs:0012 处（但是该句不执行）。如果中间包含了循环，自然会把循环执行完

  • 汇编并不是一种智能的语言，当执行add al, 255且al中数 >1 的时候，不要妄想它会自动帮你把位进到 ah 中，他只会把溢出的部分丢掉，保留低八位。（因为你现在只是对 al 进行操作，al 只有八位）

  • 在源代码中引用内存内容时，格式只能是 [index]，和(段前缀：)[bx]两种类型，且[bx]的段前缀若省略，则系统默认是ds，当然也可以显式指出。而且段前缀有且只有ds:, cs:, ss:, es:。但是如果用[index]的形式，前面不管段前缀写什么都会被当成ds:[index] 更新：si, di 也可以实现与bx一样的功能，即是说，如果想动态的访问内从，变量只能是这三个寄存器。（而且性质与bx一样）

  • syntax sugar: 偏移地址可以写作 [dx/di/si + index]，就像C语言的数组那样，而且0[dx]和[dx + 0]是等效的

  • 段寄存器，内存数据，在值想要改变的时候，必须要经过通用寄存器进行缓冲，由通用寄存器送入

  • 一般情况下，dos和其他合法程序都不会使用 0:200 ~ 0:2ff 这256个空间，他们全是零，所以我们可以在这里操作

  • trick: 因为 s: ...一定在loop s前面，所以可以断言，从s到loop的代码的执行次数，与进入s时cx的大小相同。

  • 80 86 汇编中，一个段的大小不能超过 64kb，为啥？因为偏移地址只有16位。

  • 即使你已经assume ss:stack，在后面想用ss的话还是需要预先mov ax, stack, mov ss, ax

  • 程序只是一段 000001110110001010101001 这样的数据，你需要用一个start:来指明运行时从哪里开始，类似C中的main()函数。如果你不写那就默认当做从头开始了。 不过并不是说编译器会识别有个 start 关键字，而是你定义了这样一个 mark 以后，在程序结束的时候，写上end start，即由end后面的关键字钦定，让编译器知道。所以如果你乐意，也可在入口处写fuck:，最后end fuck即可

  • 教材上会这么写：”debug一个程序，因为ds=0b2d，所以程序从0b3d开始”。为啥？因为实际上加载完一个程序后，ds指到的其实是一段叫做PSP的数据，这段数据偏移地址为0~ffh，共256字节。从100h处开始是真正的程序代码。这个关系反映到偏移地址中，就是程序真正开始处的段地址为 ds+10h，即 (ds*10h + 10h) : 0

  • （难点）关于jmp short s这样的短转移的 一定要注意，机器码中存放的是位移量的大小。也就是说，把存在位置s2的jmp short s，里面的内存的机器码复制到s1处（代码中标号顺序顺序为s, s1, s2），那么在s1 处执行的jmp short s将不能正确跳转到s（会跑到s上面，因为移动的距离是固定的）。

  • 跳转，基本上可以这样分：

    1. jmp short s 是段内转移，距离八位，距离不超过 -128~127
    2. jmp near ptr s 是段内转移，距离16位，距离不超过 -32768~32767
    3. jmp far ptr s 是段间转移，距离直接写在机器代码中，两个字，低位字是目标段地址，高位字是目标偏移地址（但是这里要存疑，有可能记反了）。另外如果是call far ptr，那么就是先push cs, 再push ip

    4. 除了 加入了far ptr，其他的call 都是只存ip，而加了的话就是 push cs; push ip

    5. call symbol; ... ;symbol:...; ret这种调用函数并返回的机制，跟C语言的实现是一个意思。只不过在汇编中，call是把当前地址存到stack，ret是拿出，要小心维护。

汉诺塔问题

汉诺塔问题即梵塔问题，是一个非常有名的古代难题，具体题目如下：



“梵塔问题”源于以下的古老传说：

在世界中心贝那勒斯（印度北部的佛教圣地）的圣庙里，安放着一块黄铜板，板上插著三根细细的、镶上宝石的细针，细针像菜叶般粗，而高就像成人由手腕到肘关节的长。

当印度教的主神梵天在创造地球这个世界时，就在其中的一根针上从下到上放了半径由大到小的六十四片圆金片环，这就是有名的「梵塔」或称「汉内塔」（Towers of Hanoi）。

天神梵天要这庙的僧侣，把这些金片全部由一根针移到另外一根指定的针上，一次只能移一片，不管在什么情况下，金片环的大小次序不能变更，小金片环永远只能放在大金片环上面。

只要有一天这六十四片的金环能从指定的针上完全转移到另外指定的针上，世界末日就来到，芸芸众生、神庙一切都将消灭，万物尽入极乐世界去。

n阶梵塔移动次数:设金片数为n，则移动次数=2的n次方-1

以一秒钟移动一次计算，这需要夜以继日地搬动5800亿年！



但凡计算机相关学科的学生，在学习编程入门时都会遇到这一题,因为这是最为经典的递归问题之一，就好像一开始学几何会遇到毕达哥拉斯定理，一开始学代数会遇到鸡兔同笼问题一样，考虑到我的读者不太了解我的专业，所以有以上说明。

不过我会突然跑来写这篇东西是非常羞耻的。。。当初大一刚入学学C语言，用的谭浩强的教材，才开始介绍程序结构的时候就冒出了这一题，当时的我完全没有计算机相关基础知识，以为会操作Windows会下两个盗版软件就是会用电脑，真是naive。。。于是乎就被这题吊打了。我还在思考什么整型浮点型，记忆1和1.0,3/2和3/2.0有啥区别，你突然就冒出一个递归算法不是搞我吗。。。学校的管理态度是放任自流的，但是平时的教学又是标准的985要求，搞得好像刚接触C语言的小白理解递归是非常自然事情。后来这题就一直成了我的心理阴影。这两天反思一下自己，想算算以前的账，所以把以前的问题找出来看一看。
话说回来，我以前一直搞不定编程，大概是是因为没有分清理科和工科的区别，老是把自己当成一个搞科学研究的人，力求想清楚遇到的每一个细节，力求严谨的推导，然后就死得很惨了。。。工科的第一要义是实用性，理论虽然重要但是并不是重点。编程尤甚，不会也硬着头皮写，写完拿去跑，不要提心吊胆，看着错误一点一点调试，慢慢就会了。

现在再来看这题，那真是轻松愉快，跟切菜一样（经历了两年还是有些成长的）。。。

思路

对于n个盘子，要借助B柱把他们从A柱搬到C柱，standardization就是先把前 n - 1 个小盘子借助C柱搬到B柱，然后把此时A柱剩下的最大的盘子搬到C柱，再把B柱上的n - 1个盘子借助A柱搬到C柱。递归的不变量在于，对于前n - 1个盘子，也可以先不管第n - 1个盘子，先搬前n - 2个，对于前n - 2个盘子，也可以不管第n - 2个盘子，先搬前n - 3个盘子。。。这样一直追溯下次，就可以一直追溯到只剩一个盘子的时候，只剩一个盘子那直接从A搬到C就完事。。。

递归的神奇之处就在此————按照一种哲学式的抽象思维来写代码，而不用在脑中模拟程序每一时刻的执行过程。话说我以前就是因为老是想用脑袋一行行跟着代码走，力求甚解才会觉得编程困难的。想一上来就理清所有东西是不可能的，就好像我现在学了汇编跟组成原理才算是理解了编程时那些种种int，double在机器内部是怎么一回事，所以我觉得只有头很硬，不会想太多的人才算是对编程有天赋的人=_=。
